# 일정 목적지 제거 및 장소 검색 분리 설계

## 목표

1. 새 일정 만들기 모달에서 실제 저장에 사용되지 않는 목적지 입력을 제거한다.
2. 탐색 탭 왼쪽 상단 검색은 공백 단위 토큰화와 카테고리 동의어 확장을 사용한다.
3. 일정 상세 및 커뮤니티 편집기의 장소 선택 검색은 입력한 장소명을 하나의 검색어로 취급하는 현재 동작을 유지한다.

## 현재 문제

- `SchedulePage.vue`는 목적지를 필수 입력으로 검사하지만, 일정 생성 API에는 목적지를 전달하지 않는다. 대신 설명 문구를 만드는 데만 임시로 사용한다.
- 백엔드 `PlaceService.toSearchTokens`는 모든 장소 검색 요청을 하나의 검색어로 처리한다.
- 장소 API를 탐색 탭과 일정 상세/커뮤니티가 공유하므로, 일정 관련 장소명 검색을 고치면서 탐색 탭의 토큰 검색까지 함께 사라졌다.

## 변경 설계

### 새 일정 만들기

- `draft.destination` 상태를 제거한다.
- 목적지 입력 필드를 제거한다.
- 일정 생성 가능 조건은 제목만 필수로 유지한다.
- 생성 설명은 목적지에 의존하지 않는 일반 문구로 변경한다.
- 응답 모델의 `destination` 표시는 기존처럼 여행 유형 라벨을 사용한다.

### 장소 검색 모드

`GET /api/places`에 선택적 `searchMode` 쿼리 파라미터를 추가한다.

- `searchMode=tokenized`
  - 공백 단위로 검색어를 분리한다.
  - 기존 카페·맛집·숙소·레저·관광·쇼핑·문화 동의어 및 카테고리 확장을 적용한다.
  - 탐색 탭 왼쪽 상단 검색만 이 모드를 명시한다.
- 파라미터가 없거나 지원하지 않는 값
  - 전체 입력을 하나의 검색어로 처리한다.
  - 일정 상세 및 커뮤니티 편집기의 장소명 검색은 변경 없이 이 기본 모드를 사용한다.

기본값을 현재 동작으로 두어 기존 API 소비자의 검색 의미가 조용히 바뀌지 않게 한다.

## 데이터 흐름

1. 탐색 탭이 `keyword`와 `searchMode=tokenized`를 전송한다.
2. 컨트롤러가 검색 모드를 `PlaceSearchRequest`에 담는다.
3. 서비스가 모드에 따라 토큰 검색 또는 단일 검색을 선택한다.
4. 기존 MyBatis 검색 조건과 점수 계산은 생성된 토큰 목록을 그대로 사용한다.

## 테스트

- 프론트 API 테스트: `searchMode`가 URL 쿼리에 포함되는지 검증한다.
- 탐색 페이지 테스트: 검색 요청이 `tokenized` 모드를 사용하는지 검증한다.
- 일정 생성 테스트: 목적지 입력 없이 제목과 일정 정보만으로 생성 요청이 가능함을 검증한다.
- 백엔드 서비스 테스트:
  - `tokenized` 모드가 공백 토큰과 동의어 확장을 복원하는지 검증한다.
  - 기본 모드가 전체 입력을 단일 검색어로 유지하는지 검증한다.
  - 지원하지 않는 모드는 기본 모드로 처리되는지 검증한다.

## 범위 제외

- 데이터베이스 스키마와 여행 생성 API 계약은 변경하지 않는다.
- 일정 카드의 `destination` 표시 방식은 변경하지 않는다.
- 일정 상세 및 커뮤니티 편집기의 장소 검색 UI는 변경하지 않는다.
