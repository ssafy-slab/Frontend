# Community Comment Editing and Replies Design

## Goal

커뮤니티 댓글 작성자가 자신의 댓글을 수정·삭제할 수 있게 하고, 모든 댓글에 답글을 작성할 수 있는 대댓글 기능을 제공한다. 데이터에는 실제 답변 관계를 보존하되 화면은 깊이에 상관없이 읽기 쉬운 평면형 대화로 표현한다.

## Scope

포함 범위:

- 본인 댓글 수정
- 본인 댓글 삭제
- 댓글 및 대댓글에 답글 작성
- 삭제된 부모 댓글을 포함한 대화 관계 보존
- 수정된 댓글의 `수정됨` 표시
- 댓글 목록의 트리 정렬 및 평면형 렌더링
- 관련 백엔드·프론트엔드 회귀 테스트

제외 범위:

- 댓글 좋아요
- 댓글 신고
- 댓글 알림
- 댓글 페이지네이션
- 관리자 강제 삭제

## Existing Structure

- `COMMENT.parent_comment_id`가 이미 존재하며 자기 참조 외래 키로 연결돼 있다.
- 댓글 생성 요청은 이미 선택적인 `parentCommentId`를 받을 수 있다.
- 댓글 응답에는 `parentCommentId`, `updatedAt`, `mine`이 포함된다.
- 삭제는 물리 삭제가 아니라 `status = 'DELETED'`로 변경하는 소프트 삭제다.
- 현재 조회는 `ACTIVE` 댓글만 반환하므로 삭제된 부모와 하위 대화 관계를 함께 표시할 수 없다.
- 백엔드에는 작성자 한정 삭제가 있으나 프론트엔드 API 및 UI가 연결되지 않았다.
- 댓글 수정 API와 UI는 없다.

## Data Model

기존 `COMMENT` 테이블을 유지하고 새 경로 컬럼은 추가하지 않는다.

- `parent_comment_id`: 사용자가 실제로 답변한 댓글 ID
- `status`: `ACTIVE` 또는 `DELETED`
- `updated_at`: 수정 또는 삭제 시 갱신

댓글 관계는 제한 없는 트리로 저장한다. 대댓글의 대댓글도 바로 답변한 댓글을 `parent_comment_id`로 참조한다.

## API Design

### 댓글 생성

`POST /api/community/posts/{postId}/comments`

```json
{
  "content": "답글 내용",
  "parentCommentId": 41
}
```

검증:

- 게시글이 존재해야 한다.
- 부모 댓글이 지정된 경우 같은 게시글에 속해야 한다.
- 부모 댓글이 `ACTIVE` 상태여야 한다.
- 내용은 공백 제거 후 비어 있으면 안 되며 최대 1,000자다.

### 댓글 수정

`PUT /api/community/comments/{commentId}`

```json
{
  "content": "수정된 내용"
}
```

동작:

- 로그인 사용자와 `commenter_user_id`가 같아야 한다.
- `ACTIVE` 댓글만 수정할 수 있다.
- 성공 시 갱신된 전체 댓글 목록을 반환한다.
- 다른 사용자의 댓글이면 `403`, 없거나 이미 삭제됐으면 `404`를 반환한다.

### 댓글 삭제

`DELETE /api/community/comments/{commentId}`

- 로그인 사용자와 작성자가 같아야 한다.
- `status`를 `DELETED`로 변경한다.
- 자식 댓글은 삭제하지 않는다.
- 성공 응답은 `204 No Content`다.

### 댓글 조회

`GET /api/community/posts/{postId}/comments`

응답에 다음 필드를 추가한다.

```json
{
  "commentId": 52,
  "parentCommentId": 41,
  "replyToNickname": "민수",
  "content": "좋은 의견이에요",
  "deleted": false,
  "edited": true,
  "mine": true
}
```

- `replyToNickname`: 직접 답변 대상의 닉네임. 최상위 댓글은 `null`.
- `deleted`: 삭제된 자리 표시용 댓글인지 여부.
- `edited`: `updated_at`이 `created_at`보다 뒤이고 삭제 상태가 아닌 경우.

## Deleted Comment Policy

삭제된 댓글은 두 경우로 나눈다.

### 자식 댓글이 없는 경우

- 조회 결과에서 제외한다.
- 화면에서 완전히 사라진다.
- 게시글의 댓글 수에도 포함하지 않는다.

### 자식 댓글이 있는 경우

- 대화 구조 보존을 위해 조회 결과에 남긴다.
- 응답 내용은 원문 대신 `삭제된 댓글입니다.`로 대체한다.
- `deleted = true`, `mine = false`, `edited = false`로 반환한다.
- 원래 댓글 내용은 API에 노출하지 않는다.
- 작성자 닉네임은 자리 표시용 `삭제된 댓글`로 반환한다.
- 해당 댓글을 직접 부모로 가진 답글의 답변 대상은 `@삭제된 댓글`로 표시한다.
- 삭제된 댓글에는 답글·수정·삭제 버튼을 제공하지 않는다.

## Query and Ordering

댓글 조회는 활성 댓글과 하위 답글이 존재하는 삭제 댓글을 가져온다. 서버 또는 프론트의 순수 변환 함수에서 다음 순서로 평탄화한다.

1. 최상위 댓글을 작성 순서대로 정렬한다.
2. 각 댓글의 직계 자식을 작성 순서대로 순회한다.
3. 자식의 자식을 같은 방식으로 재귀 순회한다.
4. 부모가 유실되거나 다른 게시글을 가리키는 비정상 데이터는 마지막에 작성 순서대로 배치한다.

순환 참조에 대비해 방문한 댓글 ID 집합을 사용한다. 이미 방문한 댓글은 다시 순회하지 않는다.

댓글 수는 화면에 실제로 표시되는 항목 수를 사용한다. 자식 없는 삭제 댓글은 제외하고, 자식이 있어 자리 표시로 남은 삭제 댓글은 포함한다.

## Frontend Interaction

### 공통 레이아웃

- 최상위 댓글은 기존 위치에 표시한다.
- 모든 답글은 실제 depth와 관계없이 동일한 한 단계 들여쓰기를 사용한다.
- 답글 본문 앞에 `@답변 대상`을 브랜드 색상으로 표시한다.
- 들여쓰기 크기는 데스크톱과 모바일에서 동일한 의미를 갖되 모바일 폭을 침범하지 않는 고정 간격을 사용한다.

### 답글 작성

- 모든 활성 댓글에 `답글` 버튼을 표시한다.
- 클릭하면 해당 댓글 바로 아래에 입력창을 연다.
- 입력창 상단에 `@닉네임에게 답글`을 표시한다.
- 한 번에 하나의 답글 입력창만 열린다.
- 등록 시 `parentCommentId`에 선택한 댓글 ID를 전달한다.
- 취소하면 입력 내용과 답변 대상을 초기화한다.

### 댓글 수정

- `mine = true`이고 `deleted = false`인 댓글에만 `수정` 버튼을 표시한다.
- 클릭하면 본문 영역을 인라인 입력창으로 전환한다.
- 저장과 취소 버튼을 제공한다.
- 저장 성공 시 전체 댓글 목록을 서버 응답으로 교체한다.
- 수정된 활성 댓글은 작성 시각 옆에 `수정됨`을 표시한다.

### 댓글 삭제

- `mine = true`이고 `deleted = false`인 댓글에만 `삭제` 버튼을 표시한다.
- 삭제 전 확인 대화상자를 표시한다.
- 삭제 성공 후 댓글 목록을 다시 조회한다.
- 자식이 없다면 즉시 사라지고, 자식이 있다면 삭제 자리 표시로 변경된다.

### 삭제된 댓글

- 흐린 배경과 보조 텍스트로 `삭제된 댓글입니다.`를 표시한다.
- 답글·수정·삭제 기능을 제공하지 않는다.
- 하위 답글은 계속 표시한다.

## Error Handling

- 비로그인 사용자가 작성·수정·삭제·답글을 시도하면 로그인 안내 후 로그인 화면으로 이동한다.
- `403`: `댓글 작성자만 수정하거나 삭제할 수 있습니다.`
- `404`: `댓글을 찾을 수 없거나 이미 삭제되었습니다.`
- 부모 댓글이 삭제된 뒤 답글 등록을 시도하면 `답변할 댓글이 삭제되었습니다.`를 표시하고 댓글 목록을 새로 불러온다.
- 요청 실패 시 편집·답글 입력값을 유지해 사용자가 다시 시도할 수 있게 한다.

## Backend Components

- `CommunityCommentUpdateRequest`: 수정 내용 검증 전용 DTO
- `CommunityCommentResponse`: `replyToNickname`, `deleted`, `edited` 추가
- `CommunityController`: 댓글 수정 엔드포인트 추가
- `CommunityService`: 부모 댓글 유효성 검증, 작성자 한정 수정, 삭제 상태 처리
- `CommunityMapper`: 댓글 조회 메타데이터, 부모 검증, 수정 SQL 추가
- `CommunityMapper.xml`: 삭제 자리 표시 조회 및 답변 대상 닉네임 조인

## Frontend Components

- `communityApi.ts`: 댓글 수정·삭제 API 추가 및 응답 타입 확장
- `CommunityDetailPage.vue`: 답글·수정·삭제 상태와 이벤트 처리
- 댓글 평탄화는 별도 순수 함수로 분리해 트리 정렬과 순환 방지를 독립 테스트한다.

## Testing

### Backend

- 작성자는 활성 댓글을 수정할 수 있다.
- 다른 사용자는 수정·삭제할 수 없다.
- 삭제된 댓글은 수정할 수 없다.
- 부모 댓글은 같은 게시글의 활성 댓글이어야 한다.
- 자식 없는 삭제 댓글은 조회되지 않는다.
- 자식 있는 삭제 댓글은 원문을 노출하지 않는 자리 표시로 조회된다.
- 삭제된 부모를 직접 참조하는 답글은 `replyToNickname = "삭제된 댓글"`을 반환한다.
- `edited`는 실제 수정 댓글에만 설정된다.

### Frontend

- 본인 댓글에만 수정·삭제 버튼이 보인다.
- 답글 생성 요청에 올바른 `parentCommentId`가 포함된다.
- 모든 depth의 답글이 동일한 한 단계 들여쓰기로 렌더링된다.
- 답글에 올바른 `@답변 대상`이 표시된다.
- 수정 성공 후 내용과 `수정됨` 표시가 갱신된다.
- 자식 없는 댓글 삭제 후 목록에서 사라진다.
- 자식 있는 댓글 삭제 후 자리 표시와 하위 답글이 유지된다.
- 삭제된 댓글에는 작업 버튼이 보이지 않는다.
- 순환 또는 유실된 부모 데이터에서도 렌더링이 중단되지 않는다.

## Acceptance Criteria

- 댓글 작성자는 자신의 활성 댓글만 수정·삭제할 수 있다.
- 로그인 사용자는 모든 활성 댓글에 답글을 작성할 수 있다.
- 대댓글 depth는 데이터에서 제한하지 않는다.
- 화면의 모든 답글은 동일한 한 단계 들여쓰기를 사용한다.
- 모든 답글은 직접 답변 대상을 `@닉네임`으로 표시한다.
- 자식 없는 삭제 댓글은 완전히 숨겨진다.
- 자식 있는 삭제 댓글은 원문 없이 자리만 유지하고 하위 대화는 보존된다.
- 수정 댓글에는 작성 시각 옆에 `수정됨`이 표시된다.
